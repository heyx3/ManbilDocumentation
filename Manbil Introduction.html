Manbil Introduction

Manbil is a 3D game engine designed to be an educational tool. While there are several open-source game engines already available, they are generally not designed to be taken apart and studied by students interested in learning how these engines tick. Manbil, on the other hand, is created with the goal of having very loosely-coupled systems that can be learned from the ground up, without having to understand the total complexity of the engine.

This documentation assumes the reader is familiar with C++. Familiarity with Visual Studio is also helpful, but it isn't required.

The engine source and documentation are both available on Github.

A quick reference guide for setting up the engine and making new projectss is available here.

If you have any questions, comments, or suggestions, feel free to email William Manning at manning.w@husky.neu.edu.


Development Platform

  • C++
    • Templates are OK; macros are occasionally useful but shouldn't be abused.
    • Exceptions are used very sparingly -- only when less exotic error-handling would substantially clutter otherwise simple functions (for example, the Serialization system).
    • STL is used quite a bit, though often functions that take an std::vector will also provide an overload that takes in a raw pointer to an array so that nobody is forced to use std::vectors.
  • Visual Studio 2013
    • A powerful but Windows-only IDE
    • The "Express" edition is free.
    • Other versions (Professional, Ultimate, etc.) are expensive, but free through some student programs (such as Northeastern CCIS!).
    • One quirk to note for anybody new to Visual Studio: in the IDE, files are organized into "filters". These function exactly like folders, but they are not actual folders! It is generally a good idea to make sure the folder hierarchy matches the filter hierarchy for simplicity.
  • Windows 7 or later
    • The engine code should theoretically support OSX and Linux (except the Oculus Rift stuff), but this has never been attempted, let alone tested, and there are no build configurations set up for it.
    • The engine code should also theoretically support older versions of Windows, but this has not been tested either (and Visual Studio 2013 only works on Windows 7 and later).

Coding Standards

  • Verbosity over ambiguity.
    • Names of classes, functions, parameters, etc. try to be descriptive.
    • Every class/field/function that isn't extremely straightforward (e.x. getter/setter functions) has a comment above it explaining it.
    • Visual studio uses these comments; you can hover the mouse over any item in the code and the comments directly above its declaration will appear as a tooltip.
  • The engine is generous with line breaks to provide clear separation of code based on its context.
    • For example, there are almost always two line breaks between sections in a class declaration (fields, constructors, operators, methods).
  • Max line length is around 105.
    • 105 characters is about half the width of a 1080p screen plus some wiggle room when using Consolas font size 10.
  • Declarations of class elements are broadly split into public, protected, and private sections (in that order). Each of those sections is organized as follows:
    1. Enums, typedefs, and nested classes
    2. Static elements:
      1. Static fields
      2. Static functions
    3. Member elements:
      1. Member fields
      2. Constructors
      3. Operators
      4. Member functions
    As mentioned above, these categories tend to be separated by double line breaks.
  • Spaces, not tabs.
  • Curly braces go on their own line.
  • CamelCase for almost everything, with the following exceptions:
    • Private fields of a class (but not functions!) are lowerCamelCase.
    • Constants, enum values, and macros are CAPS_WITH_UNDERSCORES.
